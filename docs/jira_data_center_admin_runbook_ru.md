# Runbook: интеграция TimeOff и Jira Data Center для администратора (RU)

## Назначение документа

Этот документ для одного специалиста, который одновременно:

1. Администратор Jira Data Center.
2. Администратор TimeOff.
3. Ответственный за непрерывность интеграции.

Цель: настроить интеграцию "от и до", а также обеспечить поддержку, даже если основной администратор недоступен.

## Результат интеграции

1. TimeOff хранит факты отсутствий и правила замещения.
2. Интеграционный worker регулярно читает `TimeOff Integration API`.
3. Jira Data Center автоматически переназначает задачи отсутствующих сотрудников на замещающих.

## Архитектура (рекомендуемая)

1. `TimeOff`:
   источник данных об отпусках и правилах замещения.
2. `Integration Worker` (отдельный сервис/скрипт):
   опрашивает TimeOff и вызывает Jira REST API.
3. `Jira Data Center`:
   принимает команды на переназначение задач.

Почему так:

1. Надежнее и проще поддержка, чем "сложная логика внутри Jira automation".
2. Легко логировать, тестировать и восстанавливать после сбоев.

## Что уже есть в этом проекте

1. Integration API:
   `GET /integration/v1/report/replacements`
2. UI для правил замещения:
   `Settings -> Replacement rules`
3. Модель правил:
   `UserReplacement`

## Предварительные требования

1. Доступ админа к TimeOff и Jira DC.
2. HTTPS между worker, TimeOff и Jira.
3. Отдельный технический пользователь в Jira для API (service account).
4. Хранилище секретов (Vault/KMS/защищенный менеджер секретов).
5. Сервер/контейнер для worker с автоперезапуском (systemd/docker/k8s).

## Шаг 1. Подготовить TimeOff

1. Включить API:
   `Settings -> API configuration -> Enable integration API`.
2. Скопировать токен.
3. Заполнить правила замещения:
   `Settings -> Replacement rules`.
4. Проверить endpoint вручную:

```bash
curl -H "Authorization: Bearer <TIMEOFF_TOKEN>" \
  "https://<TIMEOFF_HOST>/integration/v1/report/replacements?date=2026-02-12&leave_statuses=Approved"
```

Ожидается JSON с полями:

1. `data[]`
2. `selectedReplacement`
3. `candidates`

## Шаг 2. Подготовить Jira Data Center

1. Создать service account, например `svc-timeoff-reassign`.
2. Назначить минимальные права:
   Browse Projects, Assign Issues, Edit Issues (в нужных проектах).
3. Определить область задач для переназначения:
   проекты, статусы, типы задач.
4. Согласовать JQL для рабочей логики, пример:

```text
assignee = "<jiraUser>" AND statusCategory != Done
```

## Шаг 3. Маппинг пользователей

Базовый вариант:

1. Маппинг по email:
   `timeoff.user.email -> jira user`.

Рекомендуется:

1. Хранить явную таблицу соответствий (CSV/JSON/DB), если логины в Jira не совпадают с email.
2. Добавить "fallback owner" для каждого отдела/проекта.

## Шаг 4. Настроить worker (минимальная реализация)

Worker должен делать цикл:

1. Запросить TimeOff `/integration/v1/report/replacements`.
2. Для каждого отсутствующего, у кого есть `selectedReplacement`:
3. Найти задачи в Jira по JQL.
4. Переназначить задачи на замещающего.
5. Записать лог и метрики.

Ключевые Jira API вызовы:

1. Поиск задач:
   `GET /rest/api/2/search?jql=...`
2. Переназначение:
   `PUT /rest/api/2/issue/{issueKey}/assignee`

Пример тела:

```json
{ "name": "replacement.login" }
```

## Шаг 5. Режимы запуска

Рекомендуемый график:

1. Каждые 5-15 минут.
2. Дополнительно ежедневный полный прогон утром.

Идемпотентность:

1. Если задача уже у нужного замещающего, не обновлять.
2. Не трогать задачи вне agreed scope (по проектам/статусам).

## Шаг 6. Безопасность

1. Не хранить токены в репозитории.
2. Хранить секреты только в секрет-хранилище.
3. Логировать без чувствительных данных.
4. Ограничить исходящие доступы worker только на нужные хосты.

Ротация:

1. Плановая ротация токена TimeOff раз в 90 дней.
2. Ротация токена/пароля Jira service account по внутренней политике.
3. После ротации выполнить smoke-test.

## Шаг 7. Мониторинг и алерты

Минимум метрик:

1. Успешность цикла (`success/fail`).
2. Количество обработанных отсутствующих.
3. Количество переназначенных задач.
4. Ошибки Jira API (401/403/429/5xx).
5. Время выполнения цикла.

Алерты:

1. Нет успешного цикла > 30 минут.
2. Ошибка аутентификации > 3 циклов подряд.
3. Резкий рост ошибок переназначения.

## Шаг 8. Проверка после внедрения (UAT)

1. Создать тестового сотрудника с замещающим.
2. Сделать отпуск `Approved` на текущую дату.
3. Создать тестовые задачи в Jira.
4. Дождаться цикла worker.
5. Проверить:
   задача назначена на замещающего;
   лог содержит успешный шаг;
   повторный цикл не создает лишних изменений.

## Операционная поддержка

Ежедневно:

1. Проверка последнего успешного запуска.
2. Проверка ошибок 401/403/5xx.

Еженедельно:

1. Проверка актуальности правил замещения.
2. Проверка маппинга пользователей.

Ежемесячно:

1. Тест аварийного восстановления.
2. Проверка резервных учетных данных и доступа.

## Аварийные сценарии (когда "что-то случилось")

### 1. Worker не запускается

1. Проверить статус процесса/контейнера.
2. Проверить логи запуска и переменные окружения.
3. Перезапустить сервис.
4. Запустить ручной dry-run.

### 2. Ошибка 401 в TimeOff API

1. Проверить, включен ли Integration API в TimeOff.
2. Проверить актуальность токена.
3. При необходимости сгенерировать новый токен.
4. Обновить секреты worker и перезапустить.

### 3. Ошибка 401/403 в Jira API

1. Проверить права service account.
2. Проверить credential/token.
3. Проверить, что проекты/issue security доступны.

### 4. Неправильные переназначения

1. Остановить worker.
2. Проверить маппинг пользователей.
3. Проверить правила в `Replacement rules`.
4. Откатить назначения массовым скриптом по логу (если предусмотрено).
5. Запустить worker в dry-run и проверить результаты.

### 5. TimeOff недоступен

1. Перевести worker в режим "skip with alert".
2. Не выполнять слепые переназначения без источника данных.
3. После восстановления TimeOff выполнить полный прогон.

## План непрерывности (если вы в отпуске)

### Обязательно подготовить заранее

1. Назначить резервного администратора (backup owner).
2. Передать ему доступы по процедуре break-glass:
   TimeOff admin, Jira admin, secrets, хост worker.
3. Сохранить этот runbook в доступном месте (Confluence/внутренний wiki/репозиторий).
4. Подготовить "операционный пакет":
   список URL, имена сервисов, команды перезапуска, контакты ответственных.

### Минимальный пакет, который должен быть у backup owner

1. URL TimeOff/Jira/worker dashboard.
2. Где хранятся секреты и как их обновить.
3. Как перезапустить worker.
4. Как выполнить ручной smoke-test.
5. Как временно отключить интеграцию:
   выключить API в TimeOff или остановить worker.

## Регламент изменений

Любое изменение в интеграции (JQL, маппинг, области проектов):

1. Через PR/изменение в конфиге с ревью.
2. С записью в changelog.
3. С обязательным тестом на staging (если есть).
4. С post-check в production.

## Минимальный чек-лист "Готово к продакшену"

1. API TimeOff включен, токен сохранен в секрет-хранилище.
2. Правила замещения заполнены и проверены.
3. Worker работает по расписанию и пишет метрики.
4. Есть алерты и подтвержденный канал уведомлений.
5. Есть backup owner и процедура аварийного доступа.
6. Документ и контакты актуальны.

## Go-Live Checklist (финальная приемка)

Отметьте каждый пункт перед переводом в production:

1. Конфигурация
   `DRY_RUN=false` в production.
2. Конфигурация
   `AUTO_MAP_BY_EMAIL=true`, проверен `mapping-report.json` (нет `error`).
3. Конфигурация
   Задан `MAPPING_NOT_FOUND_THRESHOLD` (например `0` или `2` по вашей политике).
4. Конфигурация
   `ENABLE_AUTO_RESTORE=true`.
5. Конфигурация
   `ENABLE_WATCHERS=true`, решение по `REMOVE_WATCHER_ON_RESTORE` зафиксировано.
6. Jira
   У сервисного пользователя есть права: Assign Issues, Edit Issues, Manage Watchers, View Voters and Watchers.
7. Jira
   Проверена Notification Scheme для Watchers.
8. Безопасность
   Токены не хранятся в git, доступны только через секреты/защищенный `.env`.
9. Наблюдаемость
   Подключены алерты: нет успешного цикла, ошибки 401/403, превышение порога not_found.
10. Отчеты
    Генерируются `mapping-report.json` и `restore-report.json`.
11. Операции
    Доступен `reassignment-state.json`, понятна процедура восстановления при ручных правках.
12. Документация
    HR-инструкция и Admin runbook актуальны, backup owner подтвержден.

## Критерии приемки (UAT sign-off)

Считаем интеграцию принятой, если выполнены все сценарии:

1. Сценарий A (forward):
   сотрудник в отпуске -> задачи переназначены на замещающего.
2. Сценарий B (watchers):
   исходный сотрудник добавлен в watchers (или уже был, без дубликата).
3. Сценарий C (restore):
   после окончания отпуска задачи возвращены исходному assignee.
4. Сценарий D (manual override):
   если assignee изменен вручную, auto-restore не перетирает изменение.
5. Сценарий E (done issue):
   задачи в Done не возвращаются.
6. Сценарий F (mapping failure):
   при ошибке маппинга срабатывает ожидаемая защита/алерт.

## Пост-релизный контроль (первые 7 дней)

1. Ежедневно:
   проверить `mapping-report.json` и `restore-report.json`.
2. Ежедневно:
   проверить, что нет повторяющихся 401/403 в логах worker.
3. Ежедневно:
   проверить 1-2 реальных кейса с отпуском/возвратом задач.
4. На 7-й день:
   зафиксировать итоги, откорректировать пороги и регламент.

## Приложение: быстрые команды проверки

Проверка TimeOff endpoint:

```bash
curl -H "Authorization: Bearer <TIMEOFF_TOKEN>" \
  "https://<TIMEOFF_HOST>/integration/v1/report/replacements?date=2026-02-12"
```

Проверка Jira поиска:

```bash
curl -u "<JIRA_USER>:<JIRA_PASSWORD_OR_TOKEN>" \
  "https://<JIRA_HOST>/rest/api/2/search?jql=assignee%3DcurrentUser()%20AND%20statusCategory!=Done"
```

Проверка Jira reassignment:

```bash
curl -u "<JIRA_USER>:<JIRA_PASSWORD_OR_TOKEN>" \
  -X PUT \
  -H "Content-Type: application/json" \
  --data "{\"name\":\"replacement.login\"}" \
  "https://<JIRA_HOST>/rest/api/2/issue/PROJECT-123/assignee"
```
