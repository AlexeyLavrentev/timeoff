# Архитектура лицензирования LeavePilot

Статус: **принято** (architecture decision). Документ описывает решение по
лицензированию LeavePilot Premium и роль внешнего проекта `gnmyt/LicenseAPI`,
который рассматривался только как источник идей.

## TL;DR (решение)

1. **Сохраняем текущий офлайн-механизм** — подписанный RSA лицензионный blob в
   `TIMEOFF_LICENSE`, проверяемый локально публичным ключом при старте
   приложения. Это и есть рантайм-энфорсмент у клиента.
2. **Не поставляем LicenseAPI (или любой лицензионный сервер) как sidecar
   клиенту.**
3. `LicenseAPI` оценивался **только как вдохновение** для будущего
   **vendor-side License Portal** (внутреннего инструмента вендора).
4. Приватный ключ подписи — **офлайн / в KMS, никогда в БД и никогда у клиента.**
5. Клиент получает **только подписанный license-blob/файл**; рантайм LeavePilot
   проверяет его локально публичным ключом.

## Текущая модель (как есть)

- Лицензия — это конверт `{ payload, algorithm, signature }` (JSON или base64),
  передаётся через `TIMEOFF_LICENSE`; публичный ключ — через
  `TIMEOFF_LICENSE_PUBLIC_KEY`.
- `payload` сейчас: `{ customer, features[], expires }`.
- Проверка: канонический JSON + `RSA-SHA256` (`crypto.verify`), сравнение
  timing-safe. В production коммерческий режим **требует** RSA-подписанной
  лицензии (см. `lib/features.js`, `lib/edition/premium_loader.js`
  `assertCommercialLicense`). HMAC-SHA256 и неподписанные лицензии допускаются
  только в dev/test.
- Лицензии выпускаются вендором офлайн утилитой `bin/sign_license.js`
  (`--private-key-file`/секрет).
- Свойства: без сети, без БД, без phone-home; срок действия — через
  `payload.expires`. Отзыва, лимита мест и активации нет.

Этот механизм простой, аудируемый и устойчивый: секрет (приватный ключ) остаётся
у вендора, у клиента — только публичный ключ и подписанные данные.

## Почему LicenseAPI на стороне клиента небезопасен

Оценка проведена по исходникам `gnmyt/LicenseAPI` (Node.js/TypeScript + Express +
MongoDB + React, MIT). Ключевые проблемы для клиентского сценария:

1. **Приватный ключ подписи хранится в MongoDB** (поле `privateKey` в
   `src/models/Project.ts`). Поставив контейнер и БД клиенту, вендор отдаёт
   ключ подписи — клиент сможет самостоятельно выпускать любые лицензии и фичи.
   Это разрушает саму модель «секрет у вендора».
2. **Клиент контролирует сервер лицензий.** Любая клиент-сайд валидация, к
   которой у клиента есть и сервер, и данные, и ключ, обходится тривиально
   (правка БД, подмена ответов, открытый доступ к админке).
3. **Офлайн-режим реализован некорректно.** В `src/controller/validation.ts`
   срок офлайн-ключа задаётся как `renewalDate = now + 1 минута`, а строка с
   `offlineRenewalDays` закомментирована — то есть «офлайн»-ключ фактически
   требует переполучения каждую минуту и офлайном не является.
4. **Неоднозначность validation vs activation.** Есть только онлайн-валидация
   (живой lookup в БД с инкрементом `currentUses`), но **лимит использований не
   проверяется** (поля/энфорсмента `maxUses` нет), а активации как привязки
   лицензии к инсталляции/машине не существует. «Max uses» по факту — счётчик,
   а не ограничение.

Дополнительно (конфигурация по умолчанию): `app.use(cors())` — CORS открыт всем;
`DISABLE_SIGNUPS: false` — регистрация открыта; в `docker-compose.yml` Mongo без
аутентификации, SMTP-пароль в открытом виде. Дамп БД содержит приватные ключи
всех проектов. Всё это неприемлемо для поставки клиенту.

**Вывод:** LicenseAPI не используется как есть и не поставляется клиенту в любом
виде (sidecar/self-host). Его ценность — концепции UI/модели данных для
внутреннего портала вендора.

## Будущий LeavePilot License Portal (целевая модель)

Назначение: автоматизировать выпуск и учёт лицензий **на стороне вендора**.

- **Только vendor-side**, во внутреннем периметре (за SSO/VPN вендора), без
  публичной экспозиции.
- **Генерирует существующий формат `TIMEOFF_LICENSE`** (тот же конверт
  `{payload, algorithm:"RSA-SHA256", signature}`), чтобы клиентский верификатор
  (`lib/features.js`) не менялся.
- **Приватный ключ — офлайн или в KMS/HSM, никогда в БД портала и никогда у
  клиента.** Подпись выполняется через KMS API; БД портала хранит только
  метаданные выпущенных лицензий, не ключи.
- **Клиент получает только подписанный blob/файл.** Установка — как сейчас:
  значение в `TIMEOFF_LICENSE` (+ публичный ключ в `TIMEOFF_LICENSE_PUBLIC_KEY`).
- **Рантайм LeavePilot проверяет лицензию локально** публичным ключом при старте
  — без сети и без портала.

Концепции LicenseAPI отображаются так (подробнее — отдельный анализ):
projects → редакции/продукт; licenses → лицензии клиентов; permissions →
наши `features`; groups → тарифные бандлы фич; metadata → доп. поля payload
(`companyId`, `plan`, `seats`); offline licenses → наш основной формат.
`validationKey` в офлайн-модели не нужен.

## Ограничения (честно)

- **Офлайн-отзыв невозможен.** Раз лицензия подписана и работает локально без
  сети, отозвать её до истечения `expires` нельзя. Смягчение: более короткий
  срок + плановый ре-выпуск; полноценный CRL/отзыв требует phone-home.
- **Жёсткое ограничение мест/инсталляций требует phone-home.** Офлайн нельзя
  достоверно посчитать число активных инсталляций/пользователей. Поле `seats`
  можно положить в payload для мягкого предупреждения, но не для строгого
  энфорсмента.
- **Привязка `companyId`/домена снижает, но не исключает копирование.**
  Включение `companyId`/`domain` в payload мешает «случайному» переносу лицензии
  на другой инстанс, но офлайн-копирование на совпадающее окружение полностью
  не предотвращается. Полная защита — снова только через онлайн-активацию.
- **Откат часов (clock rollback)** влияет на офлайн-проверку срока. Смягчение:
  короткий срок, серверная сверка при наличии сети, хранение «high-water-mark»
  времени. Полная защита — phone-home.

Эти ограничения — следствие осознанного выбора в пользу простоты и приватности
ключа, а не дефект. Усиление (онлайн-активация) вводится только при явной
бизнес-необходимости (см. Phase 4).

## Дорожная карта

### Phase 1 — ручной выпуск (текущее состояние)
Лицензии выпускаются вендором вручную через `bin/sign_license.js`. Приватный
ключ — офлайн. Изменений в клиенте нет. **Действует сейчас.**

### Phase 2 — внутренний портал: автоматизация выпуска и истории
Внутренний vendor-side сервис автоматизирует то, что делает `sign_license.js`:
формы «клиент + план + срок», подпись через KMS, выдача blob/файла, учёт
истории выданных лицензий (клиент, хэш payload, срок, кто выпустил), аудит.
Формат лицензии и клиентский верификатор **не меняются**. Подпись — через KMS,
не в БД.

### Phase 3 — опциональные метаданные `companyId`/`domain`/`seats`
Расширение `payload` дополнительными полями. Клиентский верификатор расширяется
минимально (проверка соответствия `companyId`/домена), обратная совместимость —
через версию payload. `seats` — для мягких предупреждений. Снижает «случайный»
повторный экспорт лицензии.

### Phase 4 — опциональная активация/heartbeat (только если требует бизнес)
Отдельная изолированная подсистема онлайн-активации/heartbeat для реального
отзыва и жёсткого лимита мест. Вводится **только** при подтверждённой
бизнес-необходимости; по умолчанию не реализуется, т.к. ломает офлайн-простоту.

## Связанные материалы

- Рантайм-проверка лицензии: `lib/features.js`,
  `lib/edition/premium_loader.js`.
- Выпуск лицензии (вендор): `bin/sign_license.js`.
- CLI для проверки/просмотра: `bin/license.js`.
- Тарифные планы: `config/plan_presets.json`.
- Операции с лицензиями: `docs/license-operations.md`.
- Переменные окружения: `TIMEOFF_LICENSE`, `TIMEOFF_LICENSE_PUBLIC_KEY`.
