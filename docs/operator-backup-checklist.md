# Чеклист оператора: бэкап и восстановление

Этот документ — краткий контрольный список для оператора установки LeavePilot.
Главное правило: **дамп базы данных без ключа шифрования бесполезен**.

## Что обязательно входит в бэкап

| # | Что | Где | Зачем |
|---|-----|-----|-------|
| 1 | База данных | MySQL-дамп или файл SQLite | Все данные приложения |
| 2 | `CRYPTO_SECRET` (и `TIMEOFF_SECRET_KEY`, если задан) | `.env` / секрет-хранилище | Расшифровка секретов в БД (OIDC client secret) и хеши паролей |
| 3 | `SESSION_SECRET` | `.env` | Необязателен для восстановления данных: при смене все сессии просто разлогинятся |
| 4 | `TIMEOFF_LICENSE` + `TIMEOFF_LICENSE_PUBLIC_KEY` | `.env` (Premium) | Восстановление коммерческого режима без обращения к вендору |
| 5 | `config/app.json` / `config/app.redis.json` | репозиторий развёртывания | Конфигурация установки |
| 6 | `.env` целиком | секрет-хранилище | Быстрое воспроизведение окружения |

Redis бэкапить не нужно: в нём только сессии.

## Критичное предупреждение о ключе шифрования

Секреты в БД шифруются AES-256-GCM ключом, выведенным из
`TIMEOFF_SECRET_KEY` (если задан) или `CRYPTO_SECRET`:

- **Потеря ключа = зашифрованные секреты невосстановимы.** Их придётся
  вводить заново (например, OIDC client secret в настройках SSO).
- **Восстановление дампа в окружение с другим ключом не расшифрует секреты.**
  Храните дамп и ключ как единый комплект.
- **Ротация ключа не реализована.** Смена `CRYPTO_SECRET` инвалидирует ранее
  зашифрованные секреты.

Храните `CRYPTO_SECRET` в менеджере секретов с тем же уровнем защиты и
резервирования, что и сама база.

## Снятие бэкапа

### Docker Compose (MySQL)

```bash
docker compose exec db mysqldump -uroot -p"$MYSQL_ROOT_PASSWORD" --single-transaction "$DB_NAME" > backup_$(date +%F).sql
```

### SQLite (npm-установка)

```bash
sqlite3 db.development.sqlite ".backup 'backup_$(date +%F).sqlite'"
```

Файл базы нельзя просто копировать во время работы приложения — используйте
`.backup` или останавливайте приложение.

## Проверка восстановимости (раз в квартал)

1. Разверните чистое окружение (compose на отдельной машине или каталоге).
2. Восстановите дамп БД.
3. Подставьте **тот же** `CRYPTO_SECRET` / `TIMEOFF_SECRET_KEY`.
4. Выполните `npm run db-update` (миграции должны пройти без ошибок).
5. Войдите под администратором.
6. Если настроен SSO — откройте настройки SSO и убедитесь, что вход через
   провайдера работает (секрет расшифровался).

Бэкап, который ни разу не восстанавливали, — это не бэкап.

## Восстановление: порядок действий

1. Поднимите БД и восстановите дамп.
2. Восстановите `.env` (минимум: `CRYPTO_SECRET`, `SESSION_SECRET`, параметры БД,
   для Premium — `TIMEOFF_LICENSE`, `TIMEOFF_LICENSE_PUBLIC_KEY`).
3. Запустите приложение той же (или более новой) версии, что снимала бэкап.
4. При обновлении версии — обязательно `npm run db-update` до открытия трафика.
5. Проверьте вход, календарь, настройки SSO.
