# Процедура при компрометации приватного ключа подписи лицензий

Документ для вендора. Определяет заранее согласованный порядок действий на
случай, если приватный ключ подписи лицензий (`license_private.pem` или ключ в
KMS) скомпрометирован или есть обоснованное подозрение на компрометацию.

Контекст: лицензии LeavePilot Premium проверяются офлайн публичным ключом
(`TIMEOFF_LICENSE_PUBLIC_KEY`). Отзыв выпущенной лицензии до истечения
`payload.expires` невозможен — поэтому смена ключа затрагивает **всех**
клиентов. См. [licensing-architecture.md](licensing-architecture.md).

## Что считается компрометацией

- приватный ключ попал в git, лог, тикет, бэкап с широким доступом;
- ключ был на машине, скомпрометированной вредоносным ПО;
- уволенный сотрудник имел неконтролируемый доступ к ключу;
- KMS-аудит показывает подпись, которую вендор не выполнял;
- обнаружена «левая» лицензия с валидной подписью.

При сомнениях — считать ключ скомпрометированным.

## Немедленные действия (день 0)

1. **Зафиксировать инцидент**: время обнаружения, как узнали, кто имел доступ.
2. **Заблокировать старый ключ**: в KMS — отключить ключ (disable, не delete);
   офлайн-файл — переместить в изолированное хранилище, доступ только для
   расследования.
3. **Сгенерировать новую пару ключей** на чистой машине:

   ```bash
   openssl genpkey -algorithm RSA -out license_private_v2.pem -pkeyopt rsa_keygen_bits:2048
   openssl rsa -in license_private_v2.pem -pubout -out license_public_key_v2.pem
   ```

4. **Инвентаризация выпущенных лицензий**: список клиентов, сроки `expires`,
   каналы связи. (После появления License Portal — выгрузка из его БД.)

## Ротация у клиентов (день 0–14)

Старые лицензии продолжают работать у клиентов до `expires` — офлайн-отзыв
невозможен. Задача — перевести всех на новый ключ раньше, чем злоумышленник
воспользуется старым.

Для каждого клиента:

1. Выпустить новую лицензию новым ключом (тот же payload, новый `expires`
   не короче прежнего).
2. Отправить клиенту **новую пару значений**: `TIMEOFF_LICENSE` +
   `TIMEOFF_LICENSE_PUBLIC_KEY` (публичный ключ v2) — по доверенному каналу.
3. Клиент заменяет обе переменные окружения и перезапускает приложение.
   Проверка: `node bin/license.js verify` (или страница диагностики).

Замена атомарна с точки зрения клиента: лицензия и ключ меняются вместе,
даунтайм — один рестарт контейнера.

## Почему это работает

Рантайм доверяет только ключу из `TIMEOFF_LICENSE_PUBLIC_KEY` конкретной
установки. Как только клиент заменил публичный ключ на v2, лицензии,
подписанные украденным ключом v1, на этой установке **перестают приниматься**.
«Отзыв» реализуется сменой якоря доверия на стороне клиента, а не CRL.

## Остаточные риски

- Клиент, который не заменил ключ, продолжает доверять v1-подписям до
  истечения своей лицензии. Митигируется активным сопровождением ротации.
- Злоумышленник со старым ключом может «лицензировать» пиратские установки,
  которые никогда не общались с вендором. Это эквивалентно обычному пиратству
  и офлайн-моделью не предотвращается (осознанный трейдофф архитектуры).

## Профилактика

- Приватный ключ — только в KMS/HSM или на офлайн-машине; никогда в git,
  тикетах, чатах, общих дисках.
- Доступ к подписи — минимальному кругу лиц, с аудитом (KMS API журналирует
  каждую операцию подписи).
- Срок лицензий — не более 12 месяцев: ограничивает окно ущерба.
- Раз в год — учебная ротация на тестовом клиенте, чтобы процедура была
  отработана.
