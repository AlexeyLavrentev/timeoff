# FAQ для пользователей и администраторов

## Что выбрать: npm или Docker Compose?

Если нужно быстро посмотреть приложение на одном компьютере, берите `npm`.

Если нужен запуск ближе к рабочему серверу компании, берите `docker compose`.

## Что используется по умолчанию при `npm start`?

По умолчанию в `development` используется:

- `SQLite`
- файл `db.development.sqlite`
- хранение сессий в базе

## Что используется по умолчанию при `docker compose up`?

По умолчанию используется:

- `MySQL`
- `Redis`
- конфиг `config/app.redis.json`

## Нужно ли каждый раз запускать миграции?

После первого запуска и после обновления версии — да, нужно.

Команда:

```bash
npm run db-update
```

Или:

```bash
docker compose run --rm app npm run db-update
```

## Можно ли не редактировать никакие файлы перед тестовым запуском?

Да, если запускать через `npm` и просто посмотреть приложение локально.

Для `docker compose` обязательно нужно минимум создать `.env` и заменить секреты.

## Какие файлы чаще всего редактируют до запуска?

### Для npm

- `config/app.json`

### Для Docker Compose

- `.env`
- `config/app.redis.json`

## Как поменять язык интерфейса?

В конфиге:

- `default_language`
- `supported_languages`

Также пользователь может переключать язык в интерфейсе, если он добавлен в список поддерживаемых.

## Как отключить самостоятельную регистрацию новых пользователей?

Параметр:

```json
"allow_create_new_accounts": false
```

Для compose-сценария это уже так и настроено в `config/app.redis.json`.

## Кто становится первым администратором?

Первый пользователь, зарегистрировавший новую компанию через `/register/`, автоматически становится администратором этой компании.

Это следует из логики `register_new_admin_user`.

## Почему в Docker Compose я не вижу ссылку на регистрацию?

Потому что в `config/app.redis.json` по умолчанию:

```json
"allow_create_new_accounts": false
```

Для первого старта обычно временно включают регистрацию, создают первого администратора, а затем снова отключают её.

## Можно ли использовать Redis без Docker?

Да. Нужно:

1. установить и запустить Redis;
2. в `config/app.json` включить `sessionStore.useRedis`;
3. указать `host` и `port`.

## Можно ли использовать MySQL без Docker?

Да. Нужно:

1. поднять свой MySQL;
2. задать `DB_DIALECT=mysql`;
3. задать `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`.

## Как понять, что приложение действительно работает с MySQL, а не с SQLite?

Самая надёжная проверка:

```bash
node -e "const db=require('./lib/model/db'); console.log(db.sequelize.getDialect()); db.sequelize.close();"
```

Если вывод `mysql`, значит используется `MySQL`.

## Как понять, что Redis реально используется?

После входа в приложение:

```bash
redis-cli KEYS 'sess:*'
```

Если есть ключи, Redis работает как session store.

## Почему в Docker Compose приложение не использует `config/app.json`?

Потому что compose монтирует:

```text
./config/app.redis.json:/app/config/app.json
```

То есть внутри контейнера рабочим конфигом становится файл с уже включенным Redis.

## Нужно ли ставить SQLite отдельно?

Обычно нет. Node-зависимость `sqlite3` уже входит в проект.

## Почему приложение может работать локально без `.env`, а в Docker нет?

Потому что:

- в `development` у приложения есть fallback-секреты;
- в `production`/`staging` секреты обязательны;
- в compose они жёстко ожидаются через `.env`.

## Что делать, если хочу внедрить систему в корпоративную работу?

Минимальный разумный набор:

1. запуск через `docker compose`;
2. `MySQL`;
3. `Redis`;
4. собственные секреты в `.env`;
5. корректный `application_domain`;
6. reverse proxy с HTTPS;
7. регулярный запуск миграций при обновлениях;
8. отдельная проверка LDAP/SSO, если они нужны.
