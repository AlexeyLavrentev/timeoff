# Инструкция для HR: интеграция TimeOff с Jira Data Center (пошагово)

## Для кого эта инструкция

Этот документ написан для HR и руководителей, без технической подготовки.
По нему можно:

1. Подготовить TimeOff к интеграции.
2. Проверить, что замещения настроены правильно.
3. Передать IT минимальный набор данных для подключения Jira Data Center.

## Что будет в итоге

Когда сотрудник в отпуске, задачи в Jira Data Center будут автоматически назначаться на его замещающего.

## Простыми словами

1. В TimeOff вы ведете отпуска.
2. В TimeOff задаете правило: "Если сотрудник А отсутствует, его заменяет сотрудник Б".
3. Jira регулярно проверяет TimeOff и делает переназначение задач.

## Кто участвует

1. `HR`:
   Настраивает правила замещения и поддерживает их в актуальном состоянии.
2. `Администратор TimeOff`:
   Включает Integration API и передает токен IT.
3. `IT / администратор Jira`:
   Подключает Jira Data Center к API TimeOff.

## Перед началом (чек-лист)

1. У вас есть роль администратора в TimeOff.
2. Сотрудники и отделы в TimeOff заполнены корректно.
3. В Jira у сотрудников есть учетные записи (обычно по рабочему email).
4. Есть контакт IT, который настраивает Jira Data Center.

## Шаг 1. Включить API в TimeOff

1. Войдите в TimeOff под администратором.
2. Откройте меню настроек (иконка шестеренки).
3. Перейдите в `API configuration`.
4. Включите опцию `Enable integration API`.
5. Нажмите `Save integration API configuration`.
6. Скопируйте `API access key` (токен) и сохраните в защищенном месте.

Важно:

1. Этот токен нельзя отправлять в открытых чатах.
2. Если токен случайно раскрыт, нажмите `Regenerate token` и сообщите новый токен IT.

## Шаг 2. Настроить замещающих сотрудников

1. В TimeOff откройте `Settings -> Replacement rules`.
2. В форме укажите:
   `Employee` - кого замещают.
   `Replacement` - кто замещает.
   `Priority` - приоритет (1 = самый высокий).
3. Нажмите `Add`.
4. Повторите для всех сотрудников, для кого нужно автоматическое замещение.

Рекомендации:

1. Для ключевых сотрудников задавайте 2-3 замещающих с разным приоритетом.
2. Не назначайте сотрудника замещающим самому себе.
3. Пересматривайте правила при кадровых изменениях.

## Шаг 3. Проверить правила замещения

1. На той же странице `Replacement rules` проверьте таблицу правил.
2. При необходимости используйте фильтр по отделу (`Department`).
3. Убедитесь, что у важных сотрудников есть замещающие.

## Шаг 4. Передать IT данные для подключения Jira Data Center

Передайте IT:

1. URL TimeOff (пример: `https://timeoff.company.local`).
2. API токен из `API configuration`.
3. Подтверждение, что правила замещения заполнены.
4. Правило соответствия пользователей:
   обычно `email в TimeOff = email в Jira`.

Готовый шаблон сообщения для IT:

```text
Коллеги, прошу подключить Jira Data Center к TimeOff для автозамещения задач.

1) URL TimeOff: <вставьте URL>
2) API token: <передайте безопасным способом>
3) Endpoint для замещений:
   GET /integration/v1/report/replacements
4) Маппинг пользователей: по рабочему email
5) Правила замещения в TimeOff уже заполнены (Settings -> Replacement rules)
```

## Шаг 5. Бизнес-проверка (без техподробностей)

Сценарий проверки:

1. Выберите тестового сотрудника, у которого есть задачи в Jira.
2. Убедитесь, что для него настроен замещающий в TimeOff.
3. Оформите тестовое отсутствие (обычно на текущую дату).
4. Попросите IT запустить/дождаться цикла синхронизации.
5. Проверьте в Jira:
   задачи тестового сотрудника назначены на замещающего.

Если не сработало:

1. Проверьте, что отпуск имеет статус `Approved`.
2. Проверьте, что правило замещения создано в TimeOff.
3. Уточните у IT, совпадает ли email в TimeOff и Jira.
4. Уточните у IT, что Integration API в TimeOff включен и токен актуален.

## Частые вопросы

1. Нужно ли HR работать в Jira API?
Нет. HR работает только в TimeOff (отпуска и правила замещения).

2. Нужно ли вручную переносить задачи в Jira каждый раз?
Нет, после настройки это делает интеграция автоматически.

3. Что делать при увольнении сотрудника?
Удалите или обновите его правила замещения в `Replacement rules`.

4. Можно ли назначить несколько замещающих?
Да. Используйте разные `Priority` (1, 2, 3...).

## Регламент поддержки (рекомендуется)

1. Раз в неделю HR проверяет, что у всех ключевых сотрудников есть замещение.
2. При изменении структуры отделов обновлять правила в день изменений.
3. После регенерации API токена сразу уведомлять IT.

## Краткая памятка

1. `API configuration` - включить API и хранить токен.
2. `Replacement rules` - задать правила "кто кого замещает".
3. Если что-то не работает - сначала проверяем `Approved` отпуск, правило замещения и актуальность токена.
